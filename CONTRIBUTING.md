# Contributing to aeclib

Thank you for your interest in contributing to `aeclib`! We welcome contributions that help make this library a more accurate and comprehensive compliance engine for the AEC industry.

## How to Contribute

### 1. Identify a Domain or Provision
If you're looking to add a new building code provision or technical domain, please check the existing issues or start a discussion to ensure alignment with the library's goals.

### 2. Technical Standards
Before writing any code, please read our **[DEVELOPMENT.md](DEVELOPMENT.md)**. It contains essential information on:
- Architectural Principles (Stateless, Functional Logic)
- API Style (Architect-Friendly, Direct Parameters)
- Naming Conventions and Traceability
- Testing and Commit Standards

### 3. Development Setup
To set up your local development environment and run tests, refer to the **Getting Started** section in `DEVELOPMENT.md`.

### 4. Submitting a Pull Request
- **Strict Change Scoping:** Keep pull requests atomic and focused only on the stated objective; avoid combining unrelated stylistic cleanups or reformatting with logic changes.
- Ensure your code follows the established style and technical standards in **[STYLE.md](STYLE.md)**.
- Include unit tests for every new rule or domain.
- Use **Conventional Commits** for your commit messages.
- Provide a clear description of the technical rationale for your changes.

## Legal and Safety
By contributing to `aeclib`, you agree that your contributions will be licensed under the project's **Apache 2.0 License**. 

Please ensure that your implementations are independent and do not infringe on any copyrighted building code text. Focus on the underlying physical and mathematical intent of the provisions.
